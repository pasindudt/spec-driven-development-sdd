Spec Driven Development

From Vibe Coding to Agentic Engineering

Start the timeline

The Problem

Vibe coding

  • Fast start
  • Ambiguous requirements
  • Hidden assumptions
  • Bugs discovered late

Spec-driven development

  • Clear contract
  • Tests from specs
  • Fewer assumptions
  • Changes are explicit

How SDD works

1. Write the spec

Define behavior, inputs/outputs, and edge cases. Ambiguity is a bug.

2. Generate tests

Turn the spec into acceptance tests before implementing.

3. Implement

Write the minimum code that satisfies the spec and passes tests.

4. Verify

Run tests and review against the spec.

5. Iterate

Any behavior change starts as a spec change.

Core principles

Specs are the source of truth

If the code disagrees with the spec, the code is wrong (or the spec must be updated first).

Tests enforce the spec

Tests are executable specifications: they prevent drift.

Explicit change control

A change is a spec diff; implementation follows.

Edge cases are first-class

Write them down early so agents dont invent behavior.

Determinism over vibes

Prefer clear contracts to clever guesses.

Reviewable artifacts

Specs, tests, and code can be reviewed independently.
